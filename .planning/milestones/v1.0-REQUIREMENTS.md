# Requirements: meshek-ml

**Defined:** 2026-03-30
**Core Value:** Deliver one reproducible Colab workflow that trains a LightGBM forecast and benchmarks a newsvendor/PPO optimization — grounded in academic evidence for each method choice.

## v1 Requirements

### Approach Documentation

- [ ] **APPR-01**: Project documents which ML methods were chosen and why, citing the academic papers that support each choice
- [ ] **APPR-02**: Project documents what is explicitly deferred (dynamic pricing, E2E optimization, federated learning) and the academic rationale for deferring

### Colab Setup

- [ ] **SETUP-01**: Team member can start from a fresh Google Colab runtime and install the pipeline dependencies successfully
- [ ] **SETUP-02**: Team member can mount or access Google Drive from the notebook to read real data inputs and write workflow outputs
- [ ] **SETUP-03**: Team member can control the notebook run from one parameter cell that defines source selection, paths, dates, and seed settings

### Forecasting Pipeline

- [ ] **FORE-01**: User can generate synthetic data and train one LightGBM forecasting model end-to-end in Colab
- [ ] **FORE-02**: Pipeline uses a time-based train/validation split rather than a random split
- [ ] **FORE-03**: Pipeline reports evaluation metrics (MAE, RMSE, WMAPE, pinball loss) in notebook output
- [ ] **FORE-04**: Synthetic and real data share one forecasting code path via strict schema (`date`, `merchant_id`, `product`, `quantity`)
- [ ] **FORE-05**: Real data fails fast with clear messages when required columns are missing, malformed, or unusable

### Optimization Baseline

- [ ] **OPT-01**: User can run a newsvendor baseline ordering policy against simulated demand from the notebook
- [ ] **OPT-02**: User can train a PPO agent on the perishable inventory environment from the notebook
- [ ] **OPT-03**: Notebook benchmarks PPO vs newsvendor with fill rate, waste rate, and stockout metrics

## v2 Requirements

### Forecast-Optimization Integration

- **INTG-01**: E2E forecast-optimize with embedded newsvendor structure (Paper 5, Liao et al. E2E-PIL approach)
- **INTG-02**: Dynamic pricing as coupled action variable alongside ordering quantity

### Advanced Optimization

- **ADVOPT-01**: Dual-agent architecture for pricing + inventory at different timescales (Paper 3, Zheng et al.)
- **ADVOPT-02**: Attention-based interpretability for demand driver analysis (Papers 1, 6, 7)

### Federated Learning

- **FED-01**: Flower-based federated training across heterogeneous merchants (FedAvg → FedProx)
- **FED-02**: Merchant clustering by metadata for personalized federated models

### Workflow Extensions

- **FLOW-01**: Save reusable trained model artifact to durable output location
- **FLOW-02**: Export sample predictions for offline inspection
- **FLOW-03**: Save lightweight run manifest with parameters, split details, and metrics
- **FLOW-04**: Compare LightGBM against naive baseline forecast
- **FLOW-05**: Support additional forecasting models beyond LightGBM

### Data Flexibility

- **FLEX-01**: Map arbitrary source column names into the canonical schema
- **FLEX-02**: Aggregate transaction-level data into daily merchant-product demand

## Out of Scope

| Feature | Reason |
|---------|--------|
| Federated learning training in Colab | Separate milestone; needs stable single-merchant pipeline first |
| Dynamic pricing as action variable | All papers confirm coupling, but increases complexity; price recorded in schema for v2 |
| E2E forecast-optimize training | Paper 5 shows it's better, but two-stage is right v1 starting point |
| Real-time sensor data (IoT/RFID) | MARIOD's scope; irrelevant for small boutique merchants |
| Multi-echelon supply chain | Single-store focus for greengrocers |
| Flexible schema mapping in v1 | Strict-schema requirement must stabilize first |
| Transaction-level aggregation in v1 | Assumes a prepared daily sales table already exists |
| Heavy widget-based notebook UI | Adds maintenance complexity without helping the core workflow |
