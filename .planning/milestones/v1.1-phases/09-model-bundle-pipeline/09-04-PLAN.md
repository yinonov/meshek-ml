    The section MUST cover, in order:

    1. **Prerequisites** — authenticated `gcloud` as an operator principal with write access on `gs://meshek-prod-models`; `.venv` activated locally so `python -m meshek_ml.recommendation.cli_train` is importable; `bootstrap-cloudrun.sh` has been run at least once on the target project so the models bucket exists (D-04).

    2. **The three-command flow (D-18)** — literal code block:
       ```bash
       ./scripts/bootstrap-cloudrun.sh          # one-time, idempotent (D-04)
       ./scripts/train-and-publish-model.sh     # train + upload to gs://meshek-prod-models
       ./scripts/deploy-cloudrun.sh             # redeploy with both FUSE mounts (D-06)
       ```
       State the expected outcome: `/health` returns 200 with `model_loaded: true` (Phase 8 D-10), and `POST /recommend` for a merchant with ≥14 days of seeded sales returns `reasoning_tier: "ml_forecast"`.

    3. **Reproducibility expectations (D-11)** — running the train script twice with the same seed produces bundles whose `feature_cols` list is byte-identical and whose `residual_std` matches to within 1e-6. This is enforced by `tests/recommendation/test_model_bundle.py::test_deterministic_rerun`. The default seed is 42 (`MESHEK_TRAIN_SEED`). Defaults: 20 merchants, 180 days.

    4. **Local / dry-run modes** — document the two alternate execution modes from Plan 02:
       - `LOCAL_ONLY=1 ./scripts/train-and-publish-model.sh` — trains and writes `models/lightgbm_v1.bundle` locally, skips upload. Use for smoke tests before committing to a production upload.
       - `DRY_RUN=1 ./scripts/train-and-publish-model.sh` — prints the `gcloud storage cp` command without executing it. Use to preview the upload target.

    5. **Inspecting GCS generations** — literal code block showing how to list all versions of the bundle:
       ```bash
       gcloud storage objects list "gs://meshek-prod-models/lightgbm_v1.bundle" \
         --all-versions \
         --format='table(generation,timeCreated,size)'
       ```
       Explain that each `train-and-publish-model.sh` run creates a new generation; the previous one is preserved for 90 days per the bucket lifecycle rule (D-03).

    6. **Rolling back to a prior bundle (D-22)** — literal code block showing the `#GENERATION` copy-in-place pattern from research §"Code Examples":
       ```bash
       PRIOR_GENERATION=1234567890123456   # from the list command above
       gcloud storage cp \
         "gs://meshek-prod-models/lightgbm_v1.bundle#${PRIOR_GENERATION}" \
         "gs://meshek-prod-models/lightgbm_v1.bundle" \
         --cache-control=no-cache
       ```
       Important: research §"Pitfall 5" documents that `gcloud storage restore` does NOT work for this — it only restores soft-deleted objects, not historical generations. The copy-in-place is the only supported rollback.

    7. **Forcing a model refresh (D-19)** — Cloud Run caches the FUSE mount per revision. After uploading a new bundle (or rolling back), force a new revision with a no-op update:
       ```bash
       gcloud run services update meshek-ml \
         --region me-west1 \
         --project meshek-prod
       ```
       No image rebuild required. The new revision cold-starts against the latest GCS generation.

    8. **Why objectViewer and not objectUser (D-23)** — one short paragraph: the Cloud Run service account (`meshek-ml-run@meshek-prod.iam.gserviceaccount.com`) has `roles/storage.objectViewer` on `gs://meshek-prod-models` intentionally. The service never writes to this bucket — training is offline and happens from operator credentials. Combined with `readonly=true` on the FUSE mount (D-07), this is defense-in-depth: even a service bug cannot corrupt the model bundle in production. Cite T-9-01 from 09-03-PLAN.md threat model.

    Keep the section skimmable: short prose between code blocks, no fluff. Use fenced code blocks for all commands.
