<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| handler exception → HTTP response body | Stack traces and internal strings must not leak |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-8-10 | Information Disclosure | Stack trace leakage via generic 500 | mitigate | Generic `Exception` handler returns only `{error:{code:"internal_error", message:"An unexpected error occurred", details:{request_id:<hex>}}}`. Stack trace logged server-side with `exc_info=True`. Verified by `test_500_envelope_opaque_request_id` which asserts the raised string (`"boom"`) does NOT appear in the response body. |
| T-8-11 | Information Disclosure | RequestValidationError leaks raw input values in `details` | accept | `exc.errors()` contains the field path and error message but not the raw invalid value beyond what the caller already knows. This is the standard FastAPI 422 shape and matches D-12. |
| T-8-12 | Tampering (log injection) | Hostile `path` / `method` in the structured log line | accept | JSON encoding via `json.dumps` escapes quotes/newlines. Log consumer is the operator's log aggregator, not a downstream service. |
| T-8-13 | Repudiation | No per-request correlation between client and server | mitigate | `X-Request-ID` header returned to the client AND logged server-side; operators can correlate a client-reported incident to a specific log line. |
</threat_model>

<integration_notes>
- **Envelope scope:** D-12 says success responses stay raw (no envelope). The exception handlers only fire on failure, so success responses remain the raw pydantic model. Do NOT wrap 2xx responses.
- **`RuntimeError` ambiguity:** the research §Pattern 4 code uses `"Tier 3 requires a loaded model"` substring matching. This is intentional — Phase 6's engine.py:80 raises exactly that message. If Phase 6 ever changes the message, this handler must be updated. Consider this a documented tight coupling; prefer a dedicated exception subclass in a future phase.
- **Logging config idempotency:** calling `logging.basicConfig()` multiple times is a no-op after the first call. Test fixtures create multiple `create_app()` instances per session — the module-level guard prevents reconfiguration.
</integration_notes>
