## What Was Built

### `scripts/bootstrap-cloudrun.sh`

One-time GCP setup script (D-28). Idempotent — every create operation is guarded by a `gcloud <describe> ... >/dev/null 2>&1 ||` check:

- Enables APIs: `run.googleapis.com`, `artifactregistry.googleapis.com`, `cloudbuild.googleapis.com`, `storage.googleapis.com` (always idempotent via gcloud)
- Creates Artifact Registry Docker repo `meshek` in `me-west1` (guarded by `artifacts repositories describe`)
- Creates GCS bucket `gs://meshek-prod-merchants` in `me-west1` with uniform bucket-level access, versioning ON, and a 30-day lifecycle rule on non-current versions (guarded by `storage buckets describe`)
- Creates service account `meshek-ml-run@meshek-prod.iam.gserviceaccount.com` (guarded by `iam service-accounts describe`)
- Grants `roles/storage.objectUser` on the bucket to the SA (additive IAM binding — inherently idempotent, no guard needed)
- Accepts `--project`, `--region`, `--dry-run` flags; `DRY_RUN=1` env var also supported
- `SA_EMAIL` is re-derived after flag parsing so `--project` overrides are reflected in the email address

### `scripts/deploy-cloudrun.sh`

Routine deploy script (D-26). Runs `gcloud builds submit` then `gcloud run deploy` with all D-06..D-24 flags:

- Image tagged with git short SHA: `me-west1-docker.pkg.dev/meshek-prod/meshek/meshek-ml:<sha>`
- `--execution-environment=gen2` (with `=`, not space) — required for GCS FUSE volume mounts (D-19)
- `--add-volume name=merchants-vol,type=cloud-storage,bucket=meshek-prod-merchants` (D-06)
- `--add-volume-mount volume=merchants-vol,mount-path=/var/lib/meshek/merchants` (D-06)
- `--ingress internal-and-cloud-load-balancing` (D-20)
- `--no-allow-unauthenticated` (D-21)
- `--service-account meshek-ml-run@meshek-prod.iam.gserviceaccount.com` (D-09)
- `--memory 512Mi --cpu 1 --min-instances 0 --max-instances 2 --concurrency 40 --timeout 60 --cpu-boost` (D-13..D-18)
- `--set-env-vars MESHEK_DATA_DIR=...,MESHEK_MODEL_PATH=...,MESHEK_LOG_LEVEL=info,MESHEK_API_HOST=0.0.0.0` (D-23)
- `DRY_RUN=1` / `--dry-run`: uses `emit()` helper that `printf`s instead of executes; `gcloud run services describe` URL-fetch is gated by `if [[ "${DRY_RUN}" != "1" ]]` to ensure zero network calls in dry-run mode

## Decisions Made

1. **`bootstrap-cloudrun.sh` stays separate from `deploy-cloudrun.sh`**: The bootstrap is a once-per-project operation (APIs, bucket, SA, IAM). Merging it behind an `--init` flag on the deploy script would mix day-0 setup with day-N deploy logic and complicate the deploy script's error surface. Two scripts, two responsibilities.

2. **`emit()` helper for DRY_RUN**: A single helper function avoids repeating the `if/else` pattern on every gcloud call. The URL-fetch at the end is separately guarded because it is not a deploy command but a status read — keeping it clean.

3. **`SA_EMAIL` re-derived after flag parsing**: The plan pseudocode derived `SA_EMAIL` before flag parsing, meaning `--project other-proj` would still use the default project in the email. Fixed to re-derive after parsing (Rule 1 — pre-emptive bug fix).

4. **`--execution-environment=gen2` with `=` syntax**: Per RESEARCH §Pattern 1 and the validation spec, this flag uses `=` (not space) to ensure the grep pattern `gcloud run deploy.*--execution-environment=gen2` matches on a single line.
