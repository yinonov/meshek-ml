<domain>
## Phase Boundary

Deploy the Phase 8 FastAPI/uvicorn image to Google Cloud Run in the existing
`meshek-prod` GCP project with per-merchant SQLite files persisted across
container restarts via a GCS FUSE volume mount. This phase delivers the
deploy tooling, storage persistence wiring, SQLite compatibility fix, and
a live-URL smoke test. It does NOT change the Dockerfile's build recipe,
does NOT change application business logic, and does NOT migrate storage
to Cloud SQL (that is SEED-001, a future milestone).

<decisions>
## Implementation Decisions

### GCP Project & Region
- **D-01:** Deploy into the existing `meshek-prod` GCP project; do NOT create
  a new project. The meshek dashboard already lives there.
- **D-02:** Region: `me-west1` (Tel Aviv). Lowest latency for Israeli
  merchants and matches the dashboard's region.
- **D-03:** Service name: `meshek-ml` (separate Cloud Run service from the
  `meshek` dashboard — each has its own URL and config).

### Container Registry
- **D-04:** Use Artifact Registry, not the legacy Container Registry
  (`gcr.io`). Repository: `me-west1-docker.pkg.dev/meshek-prod/meshek`.
  Image name: `meshek-ml`, tag policy = `:latest` + git short SHA.
- **D-05:** Build with `gcloud builds submit` (Cloud Build) so no local
  Docker push auth is needed. The existing `Dockerfile` at repo root is
  used verbatim; no separate Cloud Run Dockerfile.

### Storage — GCS FUSE Mount
- **D-06:** Per-merchant SQLite files persist in a dedicated GCS bucket
  `gs://meshek-prod-merchants` mounted at `/var/lib/meshek/merchants`
  inside the container via the Cloud Run native GCS FUSE volume
  integration (GA 2024, uses `--add-volume type=cloud-storage` +
  `--add-volume-mount`). No sidecar container.
- **D-07:** Bucket location: `me-west1` (same region as the service —
  single-region to keep latency and cost down).
- **D-08:** Bucket versioning ON (soft backup); lifecycle rule deletes
  non-current versions after 30 days.
- **D-09:** Service account for the Cloud Run revision is granted
  `roles/storage.objectUser` on that bucket only (least privilege). Uses
  the default compute service account OR a dedicated
  `meshek-ml-run@meshek-prod.iam.gserviceaccount.com` — prefer dedicated.

### SQLite Compatibility Fix
- **D-10:** GCS FUSE is not POSIX-complete; SQLite's default `journal_mode=WAL`
  can corrupt under FUSE. Force `journal_mode=DELETE` in `MerchantStore`
  via a `PRAGMA journal_mode=DELETE;` issued immediately after connect, BEFORE
  any writes. Apply unconditionally (not just in production) — simpler than
  env-gating and `DELETE` journal works fine on local filesystems too.
- **D-11:** Also set `PRAGMA synchronous=FULL;` (not NORMAL) to be extra
  paranoid about durability on FUSE. Same rationale: simpler to apply
  everywhere than to condition on environment.
- **D-12:** Add a regression test (`tests/storage/test_journal_mode.py`)
  asserting the journal mode is `"delete"` on a freshly opened store.

### Cloud Run Service Configuration
- **D-13:** CPU: 1, Memory: 512 MiB (matches fly.toml, matches tested
  footprint).
- **D-14:** Min instances: 0 (scale-to-zero — nearly free at merchant
  volumes of ~1 request/day).
- **D-15:** Max instances: 2 (soft cap for v1.1; the 1-request-a-day
  greengrocer workload should never saturate one instance).
- **D-16:** Concurrency: 40 (FastAPI sync routes run in a threadpool;
  default 80 is too optimistic for a 512MB instance with LightGBM loaded).
- **D-17:** Timeout: 60s. The slowest endpoint is `POST /recommend`
  with a cold model, which is fast once loaded; 60s is generous.
- **D-18:** Startup CPU boost: ON. Lifespan has to load LightGBM +
  category defaults + pooled store; boosting CPU during startup
  shortens cold-start latency without permanent cost.
- **D-19:** Execution environment: `--execution-environment=gen2`
  (required for GCS FUSE volume mounts; gen1 cannot mount volumes).

### Ingress & Auth
- **D-20:** Ingress: `--ingress=internal-and-cloud-load-balancing` —
  only the meshek dashboard (in the same VPC / project) can call it.
  External curl from the public internet is blocked. This matches
  CONTEXT.md D-15 from Phase 8 (no public auth in v1.1).
- **D-21:** Invoker IAM: grant `roles/run.invoker` to a dedicated
  `meshek-dashboard-run@meshek-prod.iam.gserviceaccount.com` service
  account (or whatever the dashboard already runs as). No
  `--allow-unauthenticated`.
- **D-22:** For smoke-testing during deploy we use
  `gcloud run services proxy meshek-ml --region me-west1` which
  tunnels through IAM as the operator's gcloud identity — no need to
  open the service publicly to test it.

### Environment Variables on the Service
- **D-23:** Set on the service at deploy time (via
  `--set-env-vars`):
    - `MESHEK_DATA_DIR=/var/lib/meshek/merchants` — matches the FUSE mount
    - `MESHEK_MODEL_PATH=/app/models/lightgbm_v1.bundle` — baked-in model
    - `MESHEK_LOG_LEVEL=info`
    - `MESHEK_API_HOST=0.0.0.0`
  Cloud Run automatically injects `PORT`; the Dockerfile CMD hardcodes
  `--port 8000` today. Phase 8.1 changes the CMD to respect `$PORT`
  (shell-form) so Cloud Run's default 8080 and Fly.io's 8000 both work.
- **D-24:** Secrets: none for v1.1. If/when Tier 3 needs a model path
  override, use `--set-secrets` pointing at Secret Manager.

### Dockerfile Changes
- **D-25:** ONE change to `Dockerfile` in Phase 8.1: swap the exec-form
  `CMD ["uvicorn", ...]` for a shell-form or a small entrypoint that
  honors `$PORT`. Options:
    - **Preferred:** shell-form `CMD uvicorn ... --port ${PORT:-8000}`
    - **Alternative:** `EXPOSE 8000` stays for Fly.io; ENTRYPOINT script
      reads `PORT` env var
  Preferred option keeps it one-liner and preserves Fly.io port 8000.

### Deploy Tooling
- **D-26:** New script at `scripts/deploy-cloudrun.sh` — single entry
  point that takes optional `--project`, `--region`, `--service`
  flags with defaults matching D-01..D-03, runs:
    1. `gcloud builds submit --tag <AR path with git short SHA>`
    2. `gcloud run deploy` with ALL the D-06..D-24 flags baked in
    3. Prints the resulting URL
- **D-27:** Script is idempotent and safe to rerun. `set -euo pipefail`,
  verifies required APIs are enabled, prompts before creating the GCS
  bucket if missing (one-time bootstrap).
- **D-28:** A companion `scripts/bootstrap-cloudrun.sh` handles first-time
  setup: enables APIs (`run`, `artifactregistry`, `cloudbuild`, `storage`),
  creates the Artifact Registry repo, creates the GCS bucket, creates the
  dedicated service account, grants IAM roles. Idempotent — safe to
  rerun.

### Smoke Test
- **D-29:** `tests/deploy/test_cloudrun_smoke.py` — guarded by env var
  `MESHEK_CLOUDRUN_SMOKE=1` and a `MESHEK_CLOUDRUN_URL=<deployed url>`
  (or uses `gcloud run services describe` to look it up). Posts
  `POST /merchants {}`, `POST /sales` with Hebrew text, `POST /recommend`,
  and asserts 201/200/200 plus schema. Same shape as the Phase 8 Docker
  smoke test. Marked `@pytest.mark.integration`.
- **D-30:** Skipped silently when the env flag is not set — does not run
  in CI or normal test passes.

### Documentation
- **D-31:** New `docs/deploy-cloudrun.md` covering:
    - One-time bootstrap (`scripts/bootstrap-cloudrun.sh`)
    - Routine deploy (`scripts/deploy-cloudrun.sh`)
    - Rollback (`gcloud run services update-traffic meshek-ml
      --to-revisions=PREVIOUS=100`)
    - How to tail logs (`gcloud run services logs read meshek-ml`)
    - How to exec / debug (`gcloud run services proxy`)
    - Cost expectations (scale-to-zero free tier)

### Testing (local)
- **D-32:** All existing Phase 8 service tests must continue to pass
  unchanged after the `PRAGMA journal_mode=DELETE` change.
- **D-33:** The Phase 8 Docker smoke test
  (`tests/service/test_docker_smoke.py`, `MESHEK_DOCKER_SMOKE=1`) must
  still pass — i.e., the shell-form `CMD` change with `${PORT:-8000}`
  must be verified inside the existing local docker flow before the
  Cloud Run deploy.

### Out of Scope (explicitly deferred)
- **D-34:** Cloud SQL / Postgres migration (SEED-001) — not in this phase.
- **D-35:** Multi-region deployment — single region for v1.1.
- **D-36:** Cloud CDN / HTTPS load balancer in front — internal-only
  ingress is enough.
- **D-37:** CI/CD automation via Cloud Build triggers — manual
  `scripts/deploy-cloudrun.sh` invocation for v1.1.
- **D-38:** Per-caller IAM or token auth beyond service-account invoker —
  see Phase 8 D-15.
- **D-39:** Custom domain on the Cloud Run URL — `run.app` default is
  fine since only the dashboard calls it.
- **D-40:** Cost budgets / billing alerts — operator task, not code.

<specifics>
## Specific Ideas

- User already pays for GCP and has a `meshek-prod` project with the meshek
  dashboard running on Cloud Run there. We reuse that project end-to-end.
- Region `me-west1` (Tel Aviv) is the correct choice for Israeli merchants.
- GCS FUSE mount is the chosen persistence strategy over Cloud SQL
  specifically because it keeps the file-per-merchant SQLite semantics
  that Phase 5 established — zero changes to `MerchantStore` beyond the
  PRAGMA. Cloud SQL migration is tracked in SEED-001.
- Must force `journal_mode=DELETE` because GCS FUSE cannot honor SQLite's
  WAL memory-mapped I/O and atomic rename semantics. This is the single
  most important gotcha in the entire phase.
- Cloud Run concurrency=40 (not the default 80) because FastAPI sync
  routes run in threadpool and 512MiB with LightGBM loaded can't sustain
  80 parallel threads.
- Scale-to-zero + min-instances=0 + internal-only ingress = effectively
  free for a 1-request-per-merchant-per-day workload.
- Preserve Fly.io target — don't delete `fly.toml`. Cloud Run becomes
  primary; Fly.io becomes documented fallback.

</specifics>
