## Summary

Phase 8.1 deploys the Phase 8 FastAPI image to Cloud Run (`me-west1`) with per-merchant SQLite files persisted via GCS FUSE. Three orthogonal changes are needed: (1) a one-line Dockerfile CMD change to honor `$PORT`, (2) two PRAGMA lines in `MerchantStore.__init__` to neutralize WAL-mode incompatibility with GCS FUSE, and (3) two new shell scripts (`bootstrap-cloudrun.sh` + `deploy-cloudrun.sh`) plus a smoke test.

The most important technical fact is that GCS FUSE does NOT implement POSIX file locking. SQLite WAL mode relies on memory-mapped I/O and `.shm`/`.wal` sidecar files with atomic rename semantics. On FUSE, these sidecar file operations can silently fail or produce corrupted database state. Forcing `journal_mode=DELETE` (plus `synchronous=FULL`) eliminates WAL mode entirely and falls back to the classic rollback journal, which requires only simple open/write/close/rename operations that GCS FUSE handles reliably for single-writer workloads (max-instances=2 with concurrency=40 means each instance processes requests serially via FastAPI's threadpool).

All `gcloud` flag names were verified against the locally installed Google Cloud SDK 554.0.0 (`gcloud run deploy --help`). The `--cpu-boost` flag is correct; there is no `--cpu-boost-startup` variant. The `gcloud run services proxy` command exists as a GA command and proxies to `localhost:8080` by default (configurable with `--port`).

**Primary recommendation:** Implement in exactly one wave: Dockerfile CMD fix + PRAGMA injection + bootstrap script + deploy script + journal mode test + smoke test + docs.

### Pattern 1: GCS FUSE Volume Mount (gcloud deploy flags)

**What:** Native Cloud Run volume mount — no sidecar, no Dockerfile changes, no gcsfuse binary.

**When to use:** Any Cloud Run service that needs durable per-container file storage in a GCS bucket.

**Exact flag incantation** [VERIFIED: gcloud run deploy --help, SDK 554.0.0]:
```bash
gcloud run deploy meshek-ml \
  --image me-west1-docker.pkg.dev/meshek-prod/meshek/meshek-ml:${GIT_SHA} \
  --region me-west1 \
  --project meshek-prod \
  --execution-environment gen2 \
  --add-volume name=merchants-vol,type=cloud-storage,bucket=meshek-prod-merchants \
  --add-volume-mount volume=merchants-vol,mount-path=/var/lib/meshek/merchants \
  ...
```

**Key constraint:** `--execution-environment gen2` is REQUIRED for Cloud Storage volume mounts. gen1 cannot mount volumes. [VERIFIED: Cloud Run docs fetched 2026-04-15]

**IAM constraint:** Service account must have `roles/storage.objectUser` on the bucket (not just `objectViewer`) because MerchantStore creates new `.sqlite` files (writes). [VERIFIED: Cloud Run docs fetched 2026-04-15]

### Pattern 2: Shell-form CMD for $PORT

**What:** Replace the exec-form CMD with shell-form so `$PORT` variable substitution works.

**Current Dockerfile line 48** (exec-form — no shell, no variable expansion):
```dockerfile
CMD ["uvicorn", "meshek_ml.service.app:create_app", "--factory", "--host", "0.0.0.0", "--port", "8000"]
```

**New shell-form** [VERIFIED: Docker shell-form behavior is standard]:
```dockerfile
CMD uvicorn meshek_ml.service.app:create_app --factory --host 0.0.0.0 --port ${PORT:-8000}
```

Shell-form invokes `/bin/sh -c "..."`, enabling `${PORT:-8000}` substitution. Fly.io does not inject `PORT`, so the `:-8000` default preserves existing Fly.io behavior. Cloud Run injects `PORT=8080`, so Cloud Run gets 8080.

The `HEALTHCHECK` on line 46 hardcodes `localhost:8000` — it must be updated to `localhost:${PORT:-8000}` or simply left as 8000 (Cloud Run ignores Docker HEALTHCHECK; it uses its own HTTP health probe configured separately).

**Recommended:** Keep HEALTHCHECK unchanged (Cloud Run ignores it); only change the CMD line.

### Pattern 3: PRAGMA Injection in MerchantStore

**Where:** `src/meshek_ml/storage/merchant_store.py`, `MerchantStore.__init__`, after the existing `PRAGMA foreign_keys = ON` and before `_apply_migrations`.

**Current lines 230-232** (the connect block):
```python
self._conn: Optional[sqlite3.Connection] = sqlite3.connect(
    str(self._path), detect_types=sqlite3.PARSE_DECLTYPES
)
self._conn.execute("PRAGMA foreign_keys = ON")
self._conn.execute("PRAGMA journal_mode = WAL")
_apply_migrations(self._conn)
```

**New lines** — replace WAL with DELETE, add FULL sync:
```python
self._conn: Optional[sqlite3.Connection] = sqlite3.connect(
    str(self._path), detect_types=sqlite3.PARSE_DECLTYPES
)
self._conn.execute("PRAGMA foreign_keys = ON")
self._conn.execute("PRAGMA journal_mode = DELETE")
self._conn.execute("PRAGMA synchronous = FULL")
_apply_migrations(self._conn)
```

**Why before _apply_migrations:** `_apply_migrations` issues `CREATE TABLE` DDL wrapped in a `with conn:` transaction. The journal mode must be set before any DML/DDL begins. [VERIFIED: sqlite3 docs — journal_mode pragma takes effect before the next write transaction]

**Why unconditionally:** D-10/D-11 locked this. DELETE journal mode works fine on local filesystems. Simpler than env-gating.

**PRAGMA user_version interaction:** `_apply_migrations` reads `PRAGMA user_version` (read-only) and then writes it inside a `with conn:` block. `PRAGMA journal_mode=DELETE` does not affect `user_version` behavior — they are orthogonal pragmas. No conflict. [VERIFIED: sqlite.org/pragma.html — journal_mode and user_version are separate pragmas]

**Idempotency:** Both PRAGMAs are idempotent. Calling `PRAGMA journal_mode=DELETE` on an already-DELETE-mode database is a no-op. Calling it on a WAL-mode database that has an existing `.wal` file will attempt WAL checkpoint and conversion; if the DB has never been opened in WAL mode (new file) this is a no-op. For any existing Fly.io WAL-mode databases, the first open will checkpoint and switch mode — safe. [VERIFIED: sqlite.org/wal.html §4]

### Pattern 4: Deploy Script Architecture

**What:** Two shell scripts — bootstrap (one-time GCP setup) and deploy (routine).

**Pattern:** `set -euo pipefail` + idempotent gcloud commands. All gcloud commands are designed to be rerunnable safely (resource-already-exists errors tolerated with `|| true` guards).

### Anti-Patterns to Avoid

- **Using `--allow-unauthenticated`:** The service is internal-only (D-20/D-21). Never expose it publicly.
- **Using `--ingress=internal` without `cloud-load-balancing`:** The correct value is `internal-and-cloud-load-balancing` (D-20) — `internal`-only would block a future load balancer in the same project. [VERIFIED: Cloud Run ingress docs fetched 2026-04-15]
- **Using exec-form CMD with $PORT:** Exec-form CMD does not invoke a shell — `$PORT` is passed as a literal string to uvicorn, not expanded. Shell-form is required.
- **Using `PRAGMA journal_mode=WAL` on GCS FUSE:** WAL creates `.shm` and `.wal` sidecar files and relies on mmap + fcntl advisory locks, both of which GCS FUSE does not implement fully. Corruption risk is real.
- **Using `gen1` execution environment:** gen1 does NOT support volume mounts. The deploy will succeed but the mount will silently not appear. Always specify `--execution-environment gen2`.
- **Granting `roles/storage.objectViewer`:** Read-only; MerchantStore creates new SQLite files (writes). Must use `roles/storage.objectUser` (D-09). [VERIFIED: Cloud Run docs fetched 2026-04-15]

```bash
#!/usr/bin/env bash
# One-time GCP bootstrap for meshek-ml Cloud Run deployment.
# Safe to rerun — all commands are idempotent.
set -euo pipefail

PROJECT="${PROJECT:-meshek-prod}"
REGION="${REGION:-me-west1}"
SA_NAME="meshek-ml-run"
SA_EMAIL="${SA_NAME}@${PROJECT}.iam.gserviceaccount.com"
BUCKET="meshek-prod-merchants"
AR_REPO="meshek"

echo "==> Enabling required APIs..."
gcloud services enable \
  run.googleapis.com \
  artifactregistry.googleapis.com \
  cloudbuild.googleapis.com \
  storage.googleapis.com \
  --project="${PROJECT}"

echo "==> Creating Artifact Registry repository (idempotent)..."
gcloud artifacts repositories create "${AR_REPO}" \
  --repository-format=docker \
  --location="${REGION}" \
  --project="${PROJECT}" \
  --description="meshek-ml container images" \
  2>/dev/null || echo "  (already exists — skipping)"

echo "==> Creating service account (idempotent)..."
gcloud iam service-accounts create "${SA_NAME}" \
  --display-name="meshek-ml Cloud Run identity" \
  --project="${PROJECT}" \
  2>/dev/null || echo "  (already exists — skipping)"

echo "==> Creating GCS bucket (idempotent)..."
gcloud storage buckets create "gs://${BUCKET}" \
  --location="${REGION}" \
  --project="${PROJECT}" \
  --uniform-bucket-level-access \
  2>/dev/null || echo "  (already exists — skipping)"

echo "==> Enabling bucket versioning..."
gcloud storage buckets update "gs://${BUCKET}" \
  --versioning \
  --project="${PROJECT}"

echo "==> Setting lifecycle rule (delete non-current after 30 days)..."
cat > /tmp/lifecycle.json <<'EOF'
{
  "rule": [{
    "action": {"type": "Delete"},
    "condition": {"numNewerVersions": 1, "age": 30}
  }]
}
EOF
gcloud storage buckets update "gs://${BUCKET}" \
  --lifecycle-file=/tmp/lifecycle.json \
  --project="${PROJECT}"

echo "==> Granting objectUser on bucket to service account..."
gcloud storage buckets add-iam-policy-binding "gs://${BUCKET}" \
  --member="serviceAccount:${SA_EMAIL}" \
  --role="roles/storage.objectUser" \
  --project="${PROJECT}"

echo "==> Bootstrap complete."
echo "    SA: ${SA_EMAIL}"
echo "    Bucket: gs://${BUCKET}"
echo "    AR: ${REGION}-docker.pkg.dev/${PROJECT}/${AR_REPO}"
```

### `scripts/deploy-cloudrun.sh`

```bash
#!/usr/bin/env bash
# Deploy meshek-ml to Cloud Run.
# Usage: ./scripts/deploy-cloudrun.sh [--project P] [--region R] [--service S]
# All flags are optional — defaults match D-01..D-03.
set -euo pipefail

PROJECT="${PROJECT:-meshek-prod}"
REGION="${REGION:-me-west1}"
SERVICE="${SERVICE:-meshek-ml}"
AR_REPO="${AR_REPO:-meshek}"
SA_EMAIL="${SA_EMAIL:-meshek-ml-run@${PROJECT}.iam.gserviceaccount.com}"
BUCKET="${BUCKET:-meshek-prod-merchants}"

# Parse optional flags
while [[ $# -gt 0 ]]; do
  case "$1" in
    --project) PROJECT="$2"; shift 2 ;;
    --region)  REGION="$2";  shift 2 ;;
    --service) SERVICE="$2"; shift 2 ;;
    *) echo "Unknown flag: $1"; exit 1 ;;
  esac
done

IMAGE_BASE="${REGION}-docker.pkg.dev/${PROJECT}/${AR_REPO}/${SERVICE}"
GIT_SHA=$(git rev-parse --short HEAD)
IMAGE_TAG="${IMAGE_BASE}:${GIT_SHA}"
IMAGE_LATEST="${IMAGE_BASE}:latest"

echo "==> Verifying required APIs are enabled..."
for api in run.googleapis.com artifactregistry.googleapis.com cloudbuild.googleapis.com storage.googleapis.com; do
  gcloud services list --enabled --project="${PROJECT}" --filter="name:${api}" \
    --format="value(name)" | grep -q "${api}" \
    || { echo "ERROR: ${api} is not enabled. Run scripts/bootstrap-cloudrun.sh first."; exit 1; }
done

echo "==> Building and pushing image via Cloud Build..."
echo "    Image: ${IMAGE_TAG}"
gcloud builds submit \
  --tag "${IMAGE_TAG}" \
  --project "${PROJECT}" \
  .

# Also tag :latest
gcloud artifacts docker tags add \
  "${IMAGE_TAG}" \
  "${IMAGE_LATEST}" \
  --project "${PROJECT}" \
  2>/dev/null || true

echo "==> Deploying to Cloud Run..."
gcloud run deploy "${SERVICE}" \
  --image "${IMAGE_TAG}" \
  --region "${REGION}" \
  --project "${PROJECT}" \
  --service-account "${SA_EMAIL}" \
  --execution-environment gen2 \
  --add-volume name=merchants-vol,type=cloud-storage,bucket="${BUCKET}" \
  --add-volume-mount volume=merchants-vol,mount-path=/var/lib/meshek/merchants \
  --cpu 1 \
  --memory 512Mi \
  --min-instances 0 \
  --max-instances 2 \
  --concurrency 40 \
  --timeout 60s \
  --cpu-boost \
  --ingress internal-and-cloud-load-balancing \
  --no-allow-unauthenticated \
  --set-env-vars "MESHEK_DATA_DIR=/var/lib/meshek/merchants,MESHEK_MODEL_PATH=/app/models/lightgbm_v1.bundle,MESHEK_LOG_LEVEL=info,MESHEK_API_HOST=0.0.0.0"

echo "==> Deploy complete."
SERVICE_URL=$(gcloud run services describe "${SERVICE}" \
  --region "${REGION}" \
  --project "${PROJECT}" \
  --format "value(status.url)")
echo "    URL: ${SERVICE_URL}"
echo "    To test: gcloud run services proxy ${SERVICE} --region ${REGION} --port 8080 --project ${PROJECT}"
```

## SQLite PRAGMA Injection Point

**File:** `src/meshek_ml/storage/merchant_store.py`

**Exact location:** `MerchantStore.__init__`, lines 227-232 (current). Replace the `PRAGMA journal_mode = WAL` line and add `synchronous`:

```python
# CURRENT (lines 227-232):
self._conn: Optional[sqlite3.Connection] = sqlite3.connect(
    str(self._path), detect_types=sqlite3.PARSE_DECLTYPES
)
self._conn.execute("PRAGMA foreign_keys = ON")
self._conn.execute("PRAGMA journal_mode = WAL")      # <-- REMOVE
_apply_migrations(self._conn)

# NEW:
self._conn: Optional[sqlite3.Connection] = sqlite3.connect(
    str(self._path), detect_types=sqlite3.PARSE_DECLTYPES
)
self._conn.execute("PRAGMA foreign_keys = ON")
self._conn.execute("PRAGMA journal_mode = DELETE")   # GCS FUSE: WAL unsafe (D-10)
self._conn.execute("PRAGMA synchronous = FULL")      # Extra durability on FUSE (D-11)
_apply_migrations(self._conn)
```

**Execution order guarantee:** The three `execute` calls are in `__init__`, before `_apply_migrations`. Since `_apply_migrations` reads `PRAGMA user_version` (read, not write) and only issues DDL if the version is below target, the journal mode is guaranteed to be set BEFORE any write transaction begins. Ordering is:
1. `connect()` — opens the file, no transaction
2. `PRAGMA foreign_keys = ON` — connection-level, no transaction
3. `PRAGMA journal_mode = DELETE` — switches mode; if previous mode was WAL, performs checkpoint; idempotent on subsequent opens
4. `PRAGMA synchronous = FULL` — connection-level, no transaction
5. `_apply_migrations()` — reads version, conditionally writes DDL

**Conflict with PRAGMA user_version:** None. `journal_mode` and `user_version` are orthogonal. `_apply_migrations` reads `PRAGMA user_version` before any write; since journal mode is already DELETE at that point, the migration DDL will use the DELETE journal correctly. [VERIFIED: sqlite.org/pragma.html — these are independent pragmas]

## Regression Test Shape

### `tests/storage/test_journal_mode.py`

```python
"""Regression test: MerchantStore forces journal_mode=DELETE (D-10/D-12).

Verifies the PRAGMA is in effect on every freshly opened store, not just
on new SQLite files. Both new and pre-existing stores are checked.
"""
from __future__ import annotations

import pytest

from meshek_ml.storage.merchant_store import MerchantStore, MerchantProfile


def _get_journal_mode(store: MerchantStore) -> str:
    """Helper: query current journal mode via the live connection."""
    row = store._conn.execute("PRAGMA journal_mode").fetchone()
    return row[0].lower()


def _get_synchronous(store: MerchantStore) -> int:
    """Helper: query PRAGMA synchronous (0=OFF, 1=NORMAL, 2=FULL, 3=EXTRA)."""
    row = store._conn.execute("PRAGMA synchronous").fetchone()
    return row[0]


def test_journal_mode_is_delete_on_new_store(data_root):
    """A brand-new MerchantStore opens with journal_mode=delete."""
    with MerchantStore("merchant_jm_test") as store:
        assert _get_journal_mode(store) == "delete", (
            "Expected journal_mode=delete, got: "
            + _get_journal_mode(store)
        )


def test_synchronous_is_full_on_new_store(data_root):
    """A brand-new MerchantStore opens with synchronous=FULL (2)."""
    with MerchantStore("merchant_sync_test") as store:
        assert _get_synchronous(store) == 2, (
            f"Expected synchronous=2 (FULL), got: {_get_synchronous(store)}"
        )


def test_journal_mode_is_delete_on_reopened_store(data_root):
    """A store opened a second time still has journal_mode=delete."""
    mid = "merchant_reopen_test"
    # First open: create profile
    with MerchantStore(mid) as store:
        store.create_profile(MerchantProfile(merchant_id=mid))

    # Second open: reopen and verify PRAGMA persists
    with MerchantStore(mid) as store:
        assert _get_journal_mode(store) == "delete"
```

**Fixture dependency:** Uses `data_root` fixture from `tests/storage/conftest.py` (already exists — monkeypatches `MESHEK_DATA_DIR` to `tmp_path`).

```python
def _json_request(url: str, data: dict | None = None) -> tuple[int, dict]:
    """HTTP request returning (status_code, json_body)."""
    body = json.dumps(data).encode() if data is not None else None
    req = urllib.request.Request(
        url,
        data=body,
        headers={"Content-Type": "application/json"},
        method="POST" if body is not None else "GET",
    )
    try:
        with urllib.request.urlopen(req, timeout=30) as resp:
            return resp.status, json.loads(resp.read())
    except urllib.error.HTTPError as exc:
        return exc.code, json.loads(exc.read())
```

## Cloud Logging Field Mapping

### Is a code change needed?

**Current JSONFormatter output fields:** `level`, `name`, `message`, `time` (plus extra fields merged from `extra=` kwargs).

**Cloud Logging auto-detected fields** [VERIFIED: Cloud Logging structured logging docs fetched 2026-04-15]:
| JSON field | Cloud Logging mapping | Status |
|---|---|---|
| `severity` | → `LogEntry.severity` | **Not present — Cloud Logging uses `level` as fallback?** |
| `message` | → log entry summary line in Logs Explorer | Present ✓ |
| `time` | → `LogEntry.timestamp` (RFC 3339 format) | Present ✓ |
| `httpRequest` | → structured HTTP request object | Not needed |

**Key finding:** Cloud Logging explicitly recognizes `severity` as the severity field, NOT `level`. The current `JSONFormatter` emits `"level": "INFO"`. Cloud Logging will still ingest the log but will NOT promote the level to the structured `LogEntry.severity` field — the entire JSON blob lands in `jsonPayload` and the severity shows as DEFAULT.

**Recommended code change (Claude's Discretion):** Add `severity` as an alias of `level` in `JSONFormatter.format()`:

```python
# In errors.py JSONFormatter.format(), add after building payload:
payload["severity"] = record.levelname   # Cloud Logging field name
# Keep "level" for backwards compatibility with existing log parsers
```

This is a 1-line addition. Without it, all log entries appear as DEFAULT severity in Cloud Logging's Logs Explorer, making it hard to filter for ERROR or WARNING. With it, severity filtering works correctly.

**Decision:** This falls under "Claude's Discretion" (CONTEXT.md). Recommend implementing the `severity` field addition — it is a 1-line change with no downside.

## Known Pitfalls

### Pitfall 1: GCS FUSE Has No POSIX File Locking
**What goes wrong:** SQLite WAL mode creates `.wal` and `.shm` sidecar files and uses `fcntl()` advisory locks to coordinate concurrent readers/writers. GCS FUSE does not implement `fcntl()` advisory locks — lock calls silently succeed without providing mutual exclusion. Two concurrent writes can corrupt the WAL.
**Why it happens:** GCS FUSE is an object-storage-backed filesystem; object storage has no lock primitives.
**How to avoid:** Use `PRAGMA journal_mode=DELETE` (D-10). DELETE journal mode uses a single `.db-journal` file with simple open/write/close semantics — no mmap, no sidecar files. [CITED: cloud.google.com/run/docs/configuring/services/cloud-storage-volume-mounts — "Cloud Storage FUSE does not provide concurrency control for multiple writes (file locking) to the same file"]
**Warning signs:** Database "disk I/O error" exceptions at runtime; `.db-wal` or `.db-shm` files accumulating in the bucket.

### Pitfall 2: gen1 Execution Environment Silently Ignores Volume Mounts
**What goes wrong:** If `--execution-environment gen2` is omitted, Cloud Run defaults to gen1 for existing services. gen1 does not support volume mounts. The deploy succeeds without error, but `/var/lib/meshek/merchants` is an empty in-container directory — merchant SQLite files are NOT persisted to GCS.
**Why it happens:** Cloud Run silently ignores unrecognized volume configuration on gen1.
**How to avoid:** Always include `--execution-environment gen2` in the deploy command (D-19). [CITED: Cloud Run docs — "Volume mounts require the second generation execution environment"]
**Warning signs:** Merchant data disappears after container restart; GCS bucket remains empty.

### Pitfall 3: $PORT Not Expanded in Exec-Form CMD
**What goes wrong:** Exec-form `CMD ["uvicorn", ..., "--port", "8000"]` passes the literal string `"8000"` — no shell runs to expand `$PORT`. Cloud Run injects `PORT=8080` and expects the container to listen on 8080, but the process listens on 8000 — Cloud Run traffic never reaches the process.
**Why it happens:** Exec-form bypasses the shell; env var expansion requires a shell.
**How to avoid:** Use shell-form `CMD uvicorn ... --port ${PORT:-8000}` (D-25). [VERIFIED: Docker documentation on exec vs shell form CMD]
**Warning signs:** Container fails health checks immediately after deploy; Cloud Run logs show "connection refused" on port 8080.

### Pitfall 4: gcloud run services proxy Requires VPC Reachability for Internal Services
**What goes wrong:** `gcloud run services proxy` tunnels through IAM but still requires a physical TCP connection to the Cloud Run service URL. For `--ingress=internal-and-cloud-load-balancing`, the service URL is only reachable from within the VPC network — the proxy does NOT create a tunnel through GCP that bypasses ingress rules.
**Why it happens:** `gcloud run services proxy` injects an auth token; it does NOT bypass network topology.
**How to avoid:** Run the smoke test from a GCE instance inside the VPC, or use a different ingress setting during testing, or use the `--ingress=internal` alternative which, counterintuitively, IS reachable via gcloud proxy from within the project because Cloud Run automatically adds Cloud Run's own infrastructure IP ranges. [CITED: gcloud run services proxy --help: "The Cloud Run service must be reachable from the machine running this command."]
**Warning signs:** Smoke test times out polling `http://localhost:{PORT}/health`; gcloud proxy exits immediately.

### Pitfall 5: Startup Cold Start with LightGBM
**What goes wrong:** FastAPI lifespan loads LightGBM model + category defaults + ProductCatalog. This can take 3-8 seconds on a cold start. With `min-instances=0`, every cold start incurs this latency before the first request gets a response. If Cloud Run's health check fires before the app is ready, the container may be killed.
**Why it happens:** Python module import + file I/O + model deserialization.
**How to avoid:** `--cpu-boost` (D-18) gives extra CPU during startup. The degraded-start contract in `create_app()` means even if the model fails to load, the container still responds — Cloud Run health checks pass. Set `--timeout 60s` (D-17) generously. [ASSUMED: LightGBM load time estimate based on Phase 8 testing showing <2s cold start in Docker; Cloud Run may be slower]
**Warning signs:** `GET /health` returns 503 on every request (degraded mode stuck); Cloud Run revision never reports healthy.

### Pitfall 6: SQLite on FUSE Has Higher I/O Latency
**What goes wrong:** Object storage round-trips are 50-200ms per operation. SQLite with `synchronous=FULL` issues multiple fsync-equivalent calls per write transaction. A single `POST /sales` (which writes rows + updates user_version) may take 200-500ms instead of <1ms locally.
**Why it happens:** GCS FUSE translates each fsync to an object upload/metadata update.
**How to avoid:** Accept the latency — the workload is 1-2 requests/merchant/day (D-14 concurrency=40 cap; max-instances=2). The 60s timeout (D-17) provides ample headroom. Consider adding a note in `docs/deploy-cloudrun.md`. [CITED: wallacesharpedavidson.nz/post/sqlite-cloudrun — "adding a user is a lot slower than getting the users due to Cloud Storage overhead"]
**Warning signs:** `POST /recommend` times out (reads are fast; writes are the bottleneck).

### Pitfall 7: MESHEK_API_PORT vs $PORT
**What goes wrong:** The existing Dockerfile sets `ENV MESHEK_API_PORT=8000`. This env var is used by the service config but NOT by the uvicorn CMD line (the CMD uses `$PORT` injected by Cloud Run). If code anywhere reads `MESHEK_API_PORT` to determine the listening port, it will see 8000 while uvicorn actually listens on 8080 (Cloud Run's `$PORT`).
**Why it happens:** Two different env vars for the same concept.
**How to avoid:** The Phase 8 code does NOT use `MESHEK_API_PORT` for binding — it is only in the Dockerfile ENV declaration. The CMD uses `$PORT` from Cloud Run. No code change needed; just document the distinction. [VERIFIED: Phase 8 CONTEXT.md D-14 — `MESHEK_API_PORT` is a config var, not the uvicorn bind port. `app.py` creates the app but does not call `uvicorn.run()`; uvicorn is started externally via CMD]

### Pitfall 8: gcloud artifacts docker tags add Requires Authentication
**What goes wrong:** The `:latest` tag is added in the deploy script using `gcloud artifacts docker tags add`. This requires Artifact Registry API enabled and the deploying user to have write access. On first run before bootstrap, this fails.
**Why it happens:** Tag management is a separate gcloud artifacts operation.
**How to avoid:** Wrap in `|| true` in the script (as shown in the skeleton above); the `:latest` tag is cosmetic. The actual deploy uses the git SHA tag. Alternatively, push both tags in `gcloud builds submit` using a `cloudbuild.yaml` with two `docker tag` steps.
