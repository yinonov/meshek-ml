# v1.1 Learnings: Merchant Order Advisor

A distilled retrospective across Phases 5–11 (28 plans, 4 days, 142 commits) with a forward-looking integration plan for the companion `meshek` monorepo. The goal of this document is not to re-tell what shipped — it is to capture what we now know that we did not know before, and to translate that into the lowest-friction path into a greengrocer's hand.

---

## Decisions

### D-01/D-02: Tier threshold at 14 distinct sale days
Tier 3 (LightGBM) activates only when a merchant has 14+ distinct sale days; below that, fall back to Tier 2 (pooled priors) or Tier 1 (category defaults).

**Rationale:** Avoids training noise on sparse series; keeps cold-start predictable.
**Source:** `06-04-SUMMARY.md`, frontmatter decisions

### D-03/D-04: Tier 1 fixed confidence 0.2 for zero-history merchants
Category defaults return a flat 0.2 confidence regardless of category quality.

**Rationale:** Cold-start claims should be structurally less confident than any evidence-based prediction. Lets downstream UX surface "we're still learning you" messaging.
**Source:** `06-02-SUMMARY.md`

### D-05/D-06: Tier 2 shrinkage formula `n / (n + 14)`, confidence linear 0.3 → 0.6
Pooled priors blend with merchant's own sparse history; confidence grows linearly over 1..13 own-history days.

**Rationale:** Mathematically principled (empirical Bayes) and intuitive to reason about in UX copy.
**Source:** `06-02-SUMMARY.md`

### D-07/D-09: Hebrew normalization is pure-stdlib, dictionary-based
NFD decomposition + combining-mark filter (U+0591..U+05C7) for niqqud; `str.maketrans` for final-letter folding (ך→כ ם→מ ן→נ ף→פ ץ→צ); `UNIT_ALIASES` dict pre-normalized at module load.

**Rationale:** No ML, no API cost, deterministic, auditable. The meshek app currently pays GPT-4o for this work (see `services/llm-engine/src/parser.py`) — this replaces it for structured fields with zero latency.
**Source:** `07-01-SUMMARY.md`

### D-10/D-11/D-12: FastAPI lifespan factory, model loaded once, state on app.state
`build_lifespan()` is a factory called at app creation time; the model lives on `app.state.ml`, never a module-level global.

**Rationale:** Testable, overridable per request (not needed yet but free), and Cloud Run cold-start friendly.
**Source:** `06-04-SUMMARY.md`

### D-13/D-14: `RecommendationResponse` shape is frozen
`{merchant_id, recommendations[{product_id, quantity, unit}], reasoning_tier, confidence_score, generated_at}`.

**Rationale:** The API's stable contract is what `meshek`'s Fastify backend will call. Freezing it now prevents churn in two repos.
**Source:** `06-01-SUMMARY.md`

### D-25: Shell-form CMD with `${PORT:-8000}` default
Single line in Dockerfile, no entrypoint script.

**Rationale:** Works on Cloud Run (PORT=8080 injected) AND Fly.io (no PORT) without branching.
**Source:** `08.1-02-SUMMARY.md`

### Two-script operator model: `bootstrap-cloudrun.sh` + `deploy-cloudrun.sh`
Day-0 provisioning is separated from day-N deploys. `DRY_RUN=1` works on both.

**Rationale:** Mixing bootstrap into deploy would conflate one-shot IAM/bucket creation with a fast feedback loop.
**Source:** `08.1-03-SUMMARY.md`, `08.1-05-SUMMARY.md`

### GCS FUSE with DELETE journal mode (not WAL) for SQLite
PRAGMAs applied unconditionally, DELETE/FULL placed between `foreign_keys=ON` and `_apply_migrations()`.

**Rationale:** WAL requires shared memory semantics FUSE does not provide; DELETE is safe everywhere and the cost is negligible at merchant-scale write rates.
**Source:** `08.1-01-SUMMARY.md`

### `objectViewer` on the models bucket (not `objectUser`)
Service never writes bundles; least privilege by role selection.

**Rationale:** Model promotion is a separate human/CI action. Runtime write access would invite accidents.
**Source:** `09-03-SUMMARY.md`

### Dual FUSE mounts: merchants (RW) + models (RO)
Two `--add-volume` / `--add-volume-mount` pairs on the same Cloud Run revision.

**Rationale:** Keeps read-mostly model bundles separate from high-churn merchant data; independent lifecycle policies (90-day non-current on models).
**Source:** `09-03-SUMMARY.md`

### `severity` is additive to structured logs, not a replacement for `level`
Keeps existing log consumers working while adopting Cloud Logging convention.

**Rationale:** Strict backwards compat at the log-sink boundary.
**Source:** `08.1-02-SUMMARY.md`

## Lessons

### Degraded start is worth the complexity
`create_app()` catches `RuntimeError` from `build_lifespan()` and yields with `app.state.ml=None`; `/health` then returns 503. Without this, a missing or corrupt model bundle on Cloud Run startup produces a 5-minute restart loop instead of a usable error page.

**Context:** Cloud Run runtime, model bundle from a separate GCS bucket — many reasons to have the service up but the model unavailable.
**Source:** `08-01-SUMMARY.md`

### Cross-phase "Decision applied" claims can be wrong
Phase 9 discovered that `MESHEK_MODELS_DIR` was documented in D-08 as set by Phase 8.1, but it had not actually been added to `ENV_VARS` in `deploy-cloudrun.sh`.

**Context:** Verifiers that only read summaries would have missed this; live DRY_RUN output caught it.
**Source:** `09-03-SUMMARY.md`

### Smoke tests drift from the API contract they claim to smoke
Phase 10 fixed four bugs in one atomic commit: import error, sales path shape, sales assertions, recommend path. The test had been merged green earlier because it was never run against the live service.

**Context:** Smoke tests must be wired into a CI step that targets the real deployed URL, not a mock.
**Source:** `10-01-SUMMARY.md`

### A single milestone audit is worth a dozen verification passes
v1.1 audit flagged 17/17 requirements as satisfied, but the round-trip through REQUIREMENTS.md checkboxes and SUMMARY frontmatter flagged several docs skew issues Phase 11 had to clean up.

**Context:** Same pattern emerged in v0.2 of the meshek repo — audits catch gaps that per-phase verifiers miss.
**Source:** `v1.1-MILESTONE-AUDIT.md`, cross-referenced to meshek `.planning/RETROSPECTIVE.md`

### Dictionary-based Hebrew parsing covers 90% of merchant vocabulary for ~0% of the cost
30 products + plural/misspelling variants + unit aliases handles typical daily sales messages. The long tail (qualifiers like "שרי", "בשל", brand names) is what still needs LLM help — but only when the dictionary miss-rate justifies the token spend.

**Context:** Relevant to `meshek`'s `services/llm-engine` which currently LLM-parses every message unconditionally.
**Source:** `07-01-SUMMARY.md`, meshek `services/llm-engine/src/parser.py` SYSTEM_PROMPT

### Two scripts, two responsibilities, one DRY helper
Both `bootstrap-cloudrun.sh` and `deploy-cloudrun.sh` share a single `emit()` / `run` helper that respects `DRY_RUN=1`. Adding a `--init` flag to deploy would have doubled the error surface.

**Context:** Operator clarity > operator convenience, when the alternative is "one script that does everything."
**Source:** `08.1-03-SUMMARY.md`

### Confidence scoring is a product surface, not a math problem
Tier 3 confidence is still a documented placeholder bounded [0.6, 0.95]. The math is less important than the shape of what the dashboard renders.

**Context:** The only caller that consumes `confidence_score` today is the future dashboard. The formula can iterate once we see the UI.
**Source:** `v1.1-MILESTONE-AUDIT.md`, tech_debt section

## Patterns

### FastAPI factory + asynccontextmanager lifespan wrapping a `build_lifespan()` factory
**When to use:** Any service where startup can partially fail and the service should still be reachable to report it. Pair with a `/health` endpoint that reads `app.state.ml` and returns 503 when None.
**Source:** `08-01-SUMMARY.md`, `06-04-SUMMARY.md`

### Module-level normalized alias dicts
Build dicts once at import time with keys already normalized; callers do a single `dict.get()`.
**When to use:** Any Hebrew/Arabic text dispatch. Avoids per-call normalization of the key, avoids per-call normalization of the table.
**Source:** `07-01-SUMMARY.md`

### Idempotent `describe || create` for cloud resources
```sh
gcloud storage buckets describe "$B" >/dev/null 2>&1 || \
  gcloud storage buckets create "$B" --location=me-west1 ...
```
**When to use:** Any bootstrap script that may be re-run. Avoids guarding with flags; the describe is free.
**Source:** `09-03-SUMMARY.md`, `08.1-03-SUMMARY.md`

### `emit()` / `run` helper for `DRY_RUN`
Every `gcloud` call routes through `run "$@"`; status-read commands are unguarded.
**When to use:** Any deploy script operators will read before trusting.
**Source:** `08.1-03-SUMMARY.md`

### Dual FUSE mount: data (RW) + models (RO)
Merchant data and model bundles have different lifecycles, backup policies, and IAM needs. Keep them separate from the start.
**When to use:** Any Cloud Run service that needs both mutable tenant state and shared static artifacts.
**Source:** `09-03-SUMMARY.md`

### Constants mirrored across modules, not imported
`training.py` defines its own `TARGET_COL` rather than importing from `forecasting.pipeline`.
**When to use:** When a downstream module's behavior must not silently change if an upstream is refactored. Trades DRY for structural independence.
**Source:** `06-03-SUMMARY.md`

### `MESHEK_MODELS_DIR` / `_data_root` re-read per call
Env vars resolved at call time, not import time.
**When to use:** Any path configuration that tests want to override via `monkeypatch.setenv` or deployments want to swap per revision.
**Source:** `06-03-SUMMARY.md`, `05-*-SUMMARY.md`

### `store_factory` as a callable, not a class reference
Lets callers inject anything (caching wrapper, must_exist=True, test double) without changing the engine API.
**When to use:** When you want Phase N to customize instantiation without touching the core module.
**Source:** `06-04-SUMMARY.md`

### Shell-form CMD with `${PORT:-8000}` default
Dockerfile runs on Cloud Run (PORT=8080) and anywhere else (no PORT, falls through to 8000) without two images.
**When to use:** Any container that targets both managed runtimes (Cloud Run, Fly.io) and local/laptop.
**Source:** `08.1-02-SUMMARY.md`

## Surprises

### 28 plans in 4 days
The v1.1 milestone shipped in 142 commits over 4 calendar days. The GSD plan-check + verifier loop kept plans small enough that most executed in single sessions (cf. meshek v0.1's 3.8min avg).

**Impact:** Estimation for the next milestone can assume roughly this cadence once the planning upfront cost is paid.
**Source:** meshek-ml STATE.md history + v1.1-MILESTONE-AUDIT.md

### SQLite WAL is incompatible with GCS FUSE
Standard "always enable WAL" advice breaks on FUSE because shared-memory semantics are not provided. DELETE journal mode is the right answer at our write rate.

**Impact:** Changed the Phase 8.1 approach — PRAGMAs became unconditional rather than env-gated.
**Source:** `08.1-01-SUMMARY.md`

### D-08 claimed an env var was set that wasn't
A prior phase summary stated that `MESHEK_MODELS_DIR` was added to deploy ENV_VARS; Phase 9 verified it had not been. Documentation drift inside the same milestone.

**Impact:** Added regression check to Plan 09-03-03. Reminder that summary frontmatter claims should be grep-verifiable, not trusted.
**Source:** `09-03-SUMMARY.md`

### Tier 3 confidence math is placeholder-forever until the dashboard ships
The formula is bounded but otherwise arbitrary. Nothing in the backend tests would tell us we have the wrong shape; only a merchant looking at a bar or a number and saying "that feels wrong" would.

**Impact:** Confidence calibration becomes a `meshek` dashboard phase, not a `meshek-ml` phase.
**Source:** `v1.1-MILESTONE-AUDIT.md` tech_debt

### Fly.io artifacts must be explicitly preserved in docs
Operators reading the new Cloud Run doc would have deleted `fly.toml` assuming Cloud Run supersedes it.

**Impact:** Dedicated "Fly.io is preserved" section in `docs/deploy-cloudrun.md`. Dual-runtime reality needs dual-runtime docs.
**Source:** `08.1-05-SUMMARY.md`

### The API smoke test had been green because it never ran
Phase 10 found that the smoke test had four bugs; tests had passed because they were not wired into Cloud Run CI — they tested a local mock.

**Impact:** Smoke tests must target live URLs, not local fixtures. See Integration Step 7 below.
**Source:** `10-01-SUMMARY.md`

## Integration Next Steps — Baking v1.1 into `meshek`

This is the forward-looking section. The ML service is deployed, stable, and has a frozen contract (D-13/D-14). Getting it into a greengrocer's hand with minimum friction is a `meshek`-repo workstream, not a `meshek-ml`-repo workstream. The ordering below is deliberate: each step is independently shippable and compounds on the previous one.

### 1. Add a thin `meshek-ml-client` package in `packages/`
Create `packages/ml-client` (TypeScript, ESM, `@meshek/ml-client`) exporting:
- `postSale(merchantId, line: string): Promise<ParsedSale>` → calls `POST /sales`
- `getRecommendation(merchantId): Promise<RecommendationResponse>` → calls `POST /recommend`
- `health(): Promise<'ok'|'degraded'>` → calls `GET /health`

Types mirror meshek-ml's `RecommendationResponse` (D-13) in `packages/types`. Zero business logic in the client — it's a typed fetch wrapper. Matches how `services/llm-engine` is already called from `packages/api`.

**Why this first:** Everything below depends on a typed client. No client = every route hand-rolls fetch.

### 2. WhatsApp inbound "morning number" — the killer merchant UX
The greengrocer sends one WhatsApp message: `"מה להזמין מחר?"` (or literally anything — treat it as intent).
- meshek's existing WhatsApp webhook routes it to a new classifier branch `intent === "daily_order_question"`.
- Backend calls `ml.getRecommendation(merchantId)`.
- Reply is a single WhatsApp message, numbered list in Hebrew, with confidence emoji:
  ```
  🛒 הזמנה למחר (ביטחון גבוה)
  1. עגבניות — 15 ק״ג
  2. מלפפון — 8 ק״ג
  3. חסה — 10 יחידות
  ...
  השב "אישור" לשליחה לספקים או ערוך בדאשבורד.
  ```
- Confidence tier → emoji: Tier 1 🌱 / Tier 2 🌿 / Tier 3 ✅ (no percentages in the WhatsApp surface — merchants do not care about 0.63 vs 0.71).

**Why this first-class:** Merchants already WhatsApp daily. No new app, no login, no settings. The UX hypothesis (memory: `merchant_context`) is that the lowest-friction path is "reply to a message I already expected."

### 3. Daily proactive push at merchant-configured hour (default 20:00)
A cron in meshek fires per merchant, calls `getRecommendation`, and sends the same formatted WhatsApp message unsolicited. Merchant replies `"אישור"` → order locked. Merchant edits → meshek's existing order editing dashboard.

**Why now, not later:** The hardest engineering (recommendation engine, Hebrew parsing, deployed API) is done. A scheduled job reading the response shape and formatting Hebrew is ~1 day of work. This is where the product actually starts helping someone.

### 4. Hybrid Hebrew parsing — dictionary first, LLM only on miss
Replace meshek's `services/llm-engine/src/parser.py` unconditional GPT-4o call with a two-stage pipeline:
1. Call `ml.parseSale(line)` (meshek-ml dictionary parser) → returns matched products with confidence.
2. For items the dictionary misses or flags as low-confidence, call GPT-4o for those items only, with a much smaller prompt (no full normalization rules, just "extract this one unrecognized item").

**Expected impact:** meshek's GPT-4o cost drops sharply on common daily traffic ("5 ק״ג עגבניות, 3 מלפפון"), stays stable on long-tail messages ("עגבניות שרי בשלות ירוקות בלי עלים"). Tracks against the cost dashboard meshek v0.2 already built.

### 5. Confidence-driven UI in the dashboard
Every recommendation line renders with:
- ✅ green dot if Tier 3 and confidence > 0.8
- 🌿 yellow dot if Tier 2 or Tier 3 with confidence 0.5–0.8
- 🌱 grey dot if Tier 1 ("we're still learning you — edit freely")

Hover/long-press shows the reason ("מבוסס על 23 ימי מכירה שלך" / "מבוסס על ממוצע הקטגוריה"). This is where the placeholder Tier 3 confidence formula (surprise #4) finally gets a real forcing function — merchants will tell us when it feels wrong.

### 6. Sales ingestion from existing orders table (no merchant change)
meshek already stores approved orders with `orderItems`. Add a background job that posts each newly-approved order as a sale to meshek-ml's `POST /sales`:
```ts
for (const item of approvedOrder.items) {
  await ml.postSale(merchantId, `${item.quantity} ${item.unit} ${item.productName}`)
}
```
This is the data flywheel: meshek-ml's Tier 3 trains on merchants meshek already has, without asking merchants for anything.

### 7. Live smoke test in CI (addressing Surprise #6)
meshek's CI already has a deployment pipeline. Add a post-deploy step that:
- Reads `MESHEK_ML_URL` from env.
- Hits `GET /health`, `POST /merchants` (test-tenant), `POST /sales`, `POST /recommend` against the live URL.
- Fails the deploy if any return unexpected shape.

Use the corrected smoke test from `meshek-ml/tests/deploy/test_cloudrun_smoke.py` (Phase 10) as the reference contract.

### 8. Merchant onboarding: zero-configuration first sale
First-time merchants in meshek hit meshek-ml's `/merchants` (auto-create on first `/sales`). No setup, no schema, no catalog import. Tier 1 category defaults run the first 14 days; Tier 2 kicks in once we see their own history; Tier 3 trains nightly once they hit 14 distinct sale days. The merchant never sees any of this — they see the UI tier dot change from 🌱 to 🌿 to ✅ over weeks.

## Sequencing Recommendation

A single new v1.2 milestone in `meshek` (not `meshek-ml`) should bundle:

1. **Steps 1, 6, 7** as the integration foundation (typed client, data flywheel, live smoke in CI) — ~3 days.
2. **Steps 2, 3** as the merchant-facing launch (WhatsApp pull + daily push) — ~3 days, gated behind a per-merchant feature flag for beta.
3. **Steps 4, 5** as the polish/cost pass (hybrid parser, confidence UI) — ~2–3 days, drives cost down and feedback loop up.
4. **Step 8** falls out for free from Steps 1 + 6.

Total: 8–9 working days on the `meshek` side. `meshek-ml` stays frozen at v1.1 unless Step 4's miss rate data suggests a dictionary expansion or Step 5's UI data suggests a confidence formula change.

## Open Questions for the Merchant Milestone

These are not blockers but are the things the v1.2 `meshek` roadmap will need to resolve:

- **Beta merchant selection.** Which 2–3 merchants opt in first? (meshek v0.3 onboarded one already — start with them.)
- **WhatsApp message budget.** Daily push at 20:00 × N merchants × 30 days = N×30 messages/month. Fits comfortably in meshek v0.2's budget alerting.
- **Rollback if Tier 3 goes bad.** If the model bundle produces bad numbers, the fastest rollback is a copy-in-place of the previous GCS bundle (documented in `docs/deploy-cloudrun.md` §9.6). Surface this to meshek operators so they do not have to touch Cloud Run.
- **Language expansion.** Dictionary currently ~30 Hebrew products. Arabic-speaking merchants exist in the target market — add to the v1.3 roadmap, not v1.2.
- **Pricing variable.** D-deferred in v1.1. If a merchant asks "should I price this cheaper to move it," we need to say "not yet" without saying "never."

## Summary

The ML capability is done. The merchant experience has not started. Every remaining piece of work to make this useful is in the `meshek` repo, not this one. The contract is frozen (D-13/D-14), the service is deployed and smoke-tested, and the hardest parts (Hebrew parsing, three-tier cold-start, Cloud Run FUSE persistence, reproducible model bundles) are behind us.

The next commit that changes a merchant's life will be a Fastify route in `packages/api`, not a LightGBM feature in `meshek_ml/`.
