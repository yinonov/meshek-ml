<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator shell → GCP IAM | bootstrap script grants `objectViewer` to the service account. Elevation beyond read is impossible from the service at runtime. |
| Cloud Run service → GCS (via FUSE) | models bucket is mounted `readonly=true`. Even a service bug cannot corrupt the bundle in place. |
| GCS bucket → Cloud Run FUSE | versioning ON + 90-day lifecycle means any accidental overwrite is recoverable via generation-pin rollback (documented in Plan 04). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-9-01 | Tampering | Models FUSE mount exposed to service | mitigate | `readonly=true` in the `--add-volume` flag (D-07) is defense-in-depth on top of `objectViewer` IAM (D-05). Service physically cannot write to `/app/models`. Research §"Pitfall 8" confirms GCS FUSE does not support random writes at all in readonly mode. Citation: cloud.google.com/run/docs/configuring/services/cloud-storage-volume-mounts. |
| T-9-02 | Elevation of Privilege | Service account could upload a poisoned bundle | mitigate | `roles/storage.objectViewer` (D-05), NOT `objectUser`. The SA has zero write capability on the models bucket. Only operator GCP principals with `storage.objectAdmin` or similar on `meshek-prod-models` can upload — enforced outside the service. |
| T-9-03 | Information Disclosure | Bucket ACL leaks bundles | accept | Uniform bucket-level access (D-03) plus IAM gating means no object-level ACL bypass; bucket is private by default. |
| T-9-04 | Repudiation | No audit trail for uploads | accept | GCS audit logging is project-wide default. Per-upload attribution available in Cloud Logging. Not load-bearing for v1.1. |
| T-9-05 | Denial of Service | FUSE cold-start latency | accept | Research §"Pitfall 3" — `--cpu-boost` already in the deploy script from Phase 8.1 D-18 mitigates this. Not a problem at v1.1 volumes. |
</threat_model>
