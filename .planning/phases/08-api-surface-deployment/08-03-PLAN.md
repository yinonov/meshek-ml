Implement API-03: `POST /sales` accepting two request shapes in a single endpoint — structured items and Hebrew free text — and persist them via `MerchantStore.write_sales`. Partial-success semantics: each failing line is reported in `skipped[]` but the request still returns 200 with `accepted_rows` as long as at least one line parsed. If ALL lines fail, return 422.

Output: the service's main ingestion endpoint, wiring the Phase 7 parser into the HTTP surface.

     class SkippedLine(BaseModel):

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HTTP client → pydantic (SalesRequest) | Untrusted `merchant_id`, `text`, `items` cross here |
| pydantic → MerchantStore | Only regex-validated `merchant_id` and length-capped `text` reach storage / parser |
| parser → MerchantStore | Canonical product slugs from the catalog; never free-form user strings |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-5-01 (inherited) | Tampering | merchant_id path traversal | mitigate | `MerchantIdStr` regex on `SalesRequest.merchant_id`; 422 fires before MerchantStore is touched. Defense-in-depth: storage layer traversal guard. |
| T-7-02 (inherited) | DoS | Oversized Hebrew input | mitigate | Phase 7 parser caps input at 2048 chars (MD-02). `SalesRequest.text` also caps at 2048 for early 422. |
| T-8-06 | Tampering | Client submits structured item with `product_id` outside the catalog | accept | MerchantStore persists whatever `product` string is in the DataFrame; downstream recommendation will simply not find it. Catalog-enforcement for structured items is out of scope for v1.1 (D-08 explicitly preserves the raw structured shape). Documented limitation. |
| T-8-07 | Integrity | Per-line parse errors silently drop sales | mitigate | Partial-success response includes `skipped[]` with `reason`; client (meshek app) is expected to surface the skipped list back to the merchant. |
</threat_model>
