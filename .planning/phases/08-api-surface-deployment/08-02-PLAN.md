<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HTTP client → pydantic layer | Untrusted `merchant_id` crosses here; rejected at 422 if it violates the regex |
| pydantic layer → MerchantStore | Only regex-validated ids reach the filesystem path builder |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-5-01 (inherited) | Tampering | merchant_id path traversal | mitigate | D-13 `Field(pattern=r"^[A-Za-z0-9_-]{1,64}$")` via `MerchantIdStr` in `CreateMerchantRequest`. Verified in `test_invalid_id_rejected_before_fs` (the test asserts zero filesystem I/O after the 422, proving the check fires pre-MerchantStore). Defense-in-depth: storage layer still enforces `_validate_merchant_id` + `Path.resolve()` parent check. |
| T-8-04 | Tampering | uuid collision letting a client overwrite another merchant's profile | accept | uuid4.hex collision probability is negligible (~2^-64 birthday bound at 2^32 merchants, far beyond v1.1 scale). `create_profile` would also fail on existing merchant via primary-key constraint. |
| T-8-05 | DoS | Oversized `display_name` | mitigate | `Field(max_length=128)` on `display_name` at pydantic layer. |
</threat_model>
