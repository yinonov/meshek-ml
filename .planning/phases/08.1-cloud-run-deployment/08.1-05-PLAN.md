Create `docs/deploy-cloudrun.md`. Seven sections, in this order:

1. **Prerequisites** — gcloud SDK installed + authenticated (`gcloud auth login`),
   operator has roles on `meshek-prod` project (owner OR the combined set listed in
   bootstrap-cloudrun.sh header comment), billing enabled, git clean working tree.
2. **One-Time Bootstrap** — run `./scripts/bootstrap-cloudrun.sh`. Explain what it
   creates (AR repo, GCS bucket `gs://meshek-prod-merchants` with versioning + 30d
   lifecycle, service account `meshek-ml-run@meshek-prod.iam.gserviceaccount.com`,
   bucket IAM binding). Note idempotency: safe to rerun.
3. **Routine Deploy** — `./scripts/deploy-cloudrun.sh` (defaults) or with
   `--project/--region/--service` overrides. Note `DRY_RUN=1` mode for pre-flight
   review. Expected output: Cloud Build logs → revision URL printed at end.
4. **Smoke Test Against Deployed URL** — two-terminal recipe:
   - Terminal 1: `gcloud run services proxy meshek-ml --region me-west1`
     (default proxy: http://localhost:8080)
   - Terminal 2:
     ```bash
     MESHEK_CLOUDRUN_SMOKE=1 \
     MESHEK_CLOUDRUN_URL=http://localhost:8080 \
     .venv/bin/python -m pytest tests/deploy/test_cloudrun_smoke.py -x -v
     ```
   Explain why: internal-ingress blocks direct `*.run.app` access; the proxy tunnels
   via the operator's IAM identity without opening the service publicly.
5. **Rollback** — single command:
   ```bash
   gcloud run services update-traffic meshek-ml \
     --to-revisions=PREVIOUS=100 \
     --region me-west1
   ```
   Explain: shifts 100% of traffic to the previous revision atomically; no image
   rebuild needed. Mention `gcloud run revisions list --service meshek-ml --region me-west1`
   for picking a specific older revision.
6. **Tail Logs / Debug** —
   ```bash
   gcloud run services logs read meshek-ml --region me-west1 --limit 100
   # stream:
   gcloud run services logs tail meshek-ml --region me-west1
   ```
   Note that JSON logs are auto-structured by Cloud Logging (`severity` field from
   Plan 02 drives the color column). Link to Logs Explorer filter
   `resource.type="cloud_run_revision" resource.labels.service_name="meshek-ml"`.
7. **Cost Expectations** —
   - `min-instances=0` → scale-to-zero → $0 idle
   - Cloud Run free tier covers ~2M requests/mo, 360k vCPU-s, 180k GiB-s
   - At ~1 request/merchant/day, 50 merchants = 1500 req/mo → well under free tier
   - GCS FUSE mount billed per bucket storage + ops; expected < $1/mo for
     50 × ~10 MB SQLite files
   - Cloud Build: first 120 min/day free; each deploy ~3 min
8. **Known Limitations** —
   - **SQLite journal mode is DELETE (not WAL)** — see D-10/D-11; WAL is unsafe on
     GCS FUSE because FUSE does not implement POSIX file locking. If merchant count
     grows past ~50 OR cross-merchant aggregations are needed OR FUSE-related
     incidents appear, migrate to Cloud SQL: [SEED-001](../.planning/seeds/SEED-001-gcp-native-storage-migration.md)
   - **max-instances=2** — intentional cap for v1.1. Do not raise without first
     verifying GCS FUSE can handle concurrent writers (SQLite DELETE journal is
     single-writer per file; each instance opens its own merchant file per request
     so concurrent distinct merchants are fine, but concurrent same-merchant writes
     from different instances will serialize via SQLite file locking — which FUSE
     approximates, not honors)
   - **Internal ingress only** — the `*.run.app` URL is not reachable from the
     public internet. Only the meshek dashboard SA (bound via `roles/run.invoker`)
     or `gcloud run services proxy` can call it
   - **No custom domain** — use the `run.app` URL; custom domain is deferred (D-39)
   - **No CI/CD** — deploys are manual `./scripts/deploy-cloudrun.sh` invocations
     for v1.1 (D-37)
   - **Fly.io is still alive** — `fly.toml` is preserved as a documented fallback;
     Phase 8.1 does not delete or deprecate it

Each section should be genuinely usable — include the literal commands an operator
will copy-paste, not placeholder text. Keep the doc self-contained (don't rely on
implicit knowledge of the scripts' internals — the scripts have their own header
comments for that).

Cross-link SEED-001 as a relative markdown link from the "Known Limitations" section.
