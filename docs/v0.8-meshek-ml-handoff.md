## Context

The current `/recommend` response ships an order quantity ("order 12") that overstates what the system actually knows. Tier 3 forecasts `realized_demand`, then `engine.py` applies a **newsvendor formula** to convert demand → order qty using assumed under/overage costs. But meshek (the consumer) has no merchant-supplied stock signal, so the order-qty layer is a leap of faith.

Separately, holidays/Ramadan/weather aren't features. Today's Tier 3 features (per `forecasting/features.py`):
- ✅ Lags (1, 7, 14, 28d), rolling mean/std (7, 14, 28d)
- ✅ Calendar: day_of_week, day_of_month, month, week_of_year, is_weekend
- ✅ Fourier annual seasonality (sin/cos of day-of-year)
- ❌ Israeli holidays as features
- ❌ Weather
- ❌ Ramadan
- ❌ Inventory / on-hand

The irony: `simulation/calendar.py` already encodes Israeli holiday dates and demand multipliers, but they're only used to *flavor synthetic labels* — the model is asked to learn holiday effects without being given the calendar.

## v1.2's purpose

Reframe the response contract to honestly reflect what the system knows, and prepare the feature pipeline so future real merchant data can flow through (holidays, weather, Ramadan). **Do not** attempt to ship a better-trained Tier 3 model. Tier 3 stays dormant until real merchant data exists (pursued as a parallel non-engineering workstream — pilot merchant outreach).

## Decisions locked (do not re-litigate in this session)

- **Defer modeling overhaul.** No M5 / Favorita / Rossmann adoption. No Tier 3 retraining loop. No real-data benchmark eval.
- **No good Israeli/ME public data exists.** Don't pretend otherwise. Lean on Tier 1/2 longer.
- **Success criterion**: honest forecaster decoupled from accuracy. Demand band, real exogenous-feature *schema* (filled by future data), per-tier signals.
- **Stock awareness**: out of scope. Demand band sidesteps it.
- **Sibling milestone**: `meshek v0.8 — Honest demand` (UX reframe + Excel ingest + per-signal reasons UI). MM-P1 (this milestone's wire contract) is the explicit synchronization point — meshek v0.8 phases M-P2/M-P3 are blocked until MM-P1 lands.

## Phases (4)

### MM-P1 — New `/recommend` response contract (wire freeze)

Replace the current response shape:
```ts
// OLD
{ merchant_id, recommendations: [{ product_id, quantity, unit }],
  reasoning_tier, confidence_score, generated_at }
```

With:
```ts
// NEW
{
  merchant_id,
  generated_at,
  recommendations: [{
    product_id,
    unit,
    predicted_demand: number,        // point estimate
    demand_lower: number,             // lower bound of band
    demand_upper: number,             // upper bound of band
    reasoning_tier: "category_default" | "pooled_prior" | "ml_forecast",
    confidence_score: number,         // [0, 1], per-line
    signals: Array<{
      name: string,                   // stable enum, e.g. "shabbat_uplift", "category_default", "pooled_prior_weight"
      contribution: number,            // signed, in demand units (or normalized — decide in this phase)
      copy_key: string                 // stable key for meshek-side i18n; e.g. "signal.shabbat_peak"
    }>
  }]
}
```

Notes:
- `quantity` is removed. The order-qty layer (newsvendor) is dropped from the contract. If kept anywhere, it's an internal computation that's no longer exposed.
- `reasoning_tier` and `confidence_score` move from response-level to per-line (this is the per-line tier work that was originally scoped for v1.2 in the old roadmap — it lands here as a side effect).
- `signals[]` is the explanation surface. Stable enum names + `copy_key` so meshek owns translation, not meshek-ml.

Update `@meshek/ml-client` (lives in meshek repo) types as part of this phase — coordinate via PR.

### MM-P2 — Exogenous feature schema

Define the input shape that *will* carry holidays / Ramadan / weather / Shabbat to Tier 3 when real data arrives. Concrete columns to add to the feature pipeline (`forecasting/features.py`):

- `is_shabbat` (bool, day-of-week derivable — universal, ship now)
- `is_friday_pre_shabbat` (bool, derivable — universal, ship now)
- `israeli_holiday` (categorical: holiday name or "none") — wire from `simulation/calendar.py`
- `israeli_holiday_proximity` (int, days to nearest holiday, signed)
- `is_ramadan` (bool) — needs Hijri calendar; new dependency (e.g. `convertdate` or `pyluach`+`hijri-converter`)
- `is_eid` (bool)
- `temperature_c` (float, optional — populated by real data only)
- `precipitation_mm` (float, optional — populated by real data only)

What's universal vs. what waits for real data:
- **Universal (ship in this phase):** all calendar-derived features (Shabbat, Hebrew holidays, Hijri holidays). Any merchant's date works.
- **Waits for real data:** weather columns. Schema accepts them; pipeline tolerates nulls.

Wire what's universal into Tier 2 priors as well (Shabbat day-of-week pattern is universal across all merchants — no need to wait for per-merchant history).

### MM-P3 — Honest tier semantics

Today only Tier 3 emits `confidence_score`. Tier 1 (category default) and Tier 2 (pooled prior) are honest defaults but the response doesn't say so to the caller.

For each tier, emit:
- `confidence_score` (Tier 1: fixed 0.2; Tier 2: linear interp 0.3→0.6 with horizon; Tier 3: residual-std-derived).
- `demand_lower` / `demand_upper` band — for Tier 1/2, derive from pooled-prior variance; for Tier 3, from residual std.
- `signals[]` with at minimum `{name: tier_name, contribution: 1.0, copy_key: "signal.tier_<n>_default"}`.

This makes the "why" surface uniform across tiers — meshek can render the same UI shape regardless of which tier produced the line.

### MM-P4 — Extend Tier 1/2 horizon

The current shrinkage formula `n / (n + 14)` caps Tier 2 at ~14 days of merchant history. Since Tier 3 is dormant longer than originally designed (waiting on pilot data), extend the Tier 2 useful horizon. Likely changes:

- Increase the shrinkage anchor from 14 to a higher value (re-tune via simulation).
- Add a "graceful degradation" path so Tier 2 confidence approaches but never reaches Tier 3's range.
- Update Tier 2 tests accordingly.
